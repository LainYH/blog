# 第2章 进程管理

## 2.1 进程与线程 ⭐⭐

### 2.1.1 进程的概念和特征

**进程：** 进程是进程实体的运行过程，是系统进行**资源分配和调度**的一个独立单位。

**进程实体（进程映像）= PCB + 程序段 + 数据段**

⚠️ **PCB（进程控制块）** 是进程存在的唯一标志！

#### 进程的特征

| 特征 | 说明 |
|------|------|
| **动态性** | 进程是程序的一次执行过程，有创建、执行、消亡的生命周期 |
| **并发性** | 多个进程实体同存于内存中，能在一段时间内同时运行 |
| **独立性** | 进程是独立获得资源和接受调度的基本单位 |
| **异步性** | 进程按各自独立的、不可预知的速度向前推进 |
| **结构性** | 进程由 PCB、程序段、数据段组成 |

### 2.1.2 进程的状态与转换 ⭐

#### 进程的五种基本状态

```
          创建态 → 就绪态 ←──────────────────┐
                     ↓                       │
                   运行态 → 阻塞态 → 就绪态 ─┘
                     ↓
                   终止态
```

| 状态 | 说明 |
|------|------|
| **创建态** | 进程正在被创建，尚未获得所需资源 |
| **就绪态** | 进程已获得除 CPU 之外的所有资源，等待 CPU |
| **运行态** | 进程正在 CPU 上执行 |
| **阻塞态（等待态）** | 进程因等待某事件（I/O 完成、信号量等）而暂停 |
| **终止态** | 进程正在从系统中消失 |

⚠️ **状态转换：**

| 转换 | 原因 |
|------|------|
| 就绪 → 运行 | 进程被调度程序选中 |
| 运行 → 就绪 | 时间片用完 / 更高优先级进程到达 |
| 运行 → 阻塞 | 请求 I/O / 申请资源失败 / 等待信号 |
| 阻塞 → 就绪 | I/O 完成 / 等待的事件发生 |

⚠️ **注意：** 阻塞态不能直接转换为运行态（必须先就绪）；运行态不能直接转换为阻塞态以外的状态（除时间片用完转就绪和终止）。

### 2.1.3 进程控制块（PCB）⭐

PCB 是操作系统用于描述进程的基本情况和运行状态的数据结构。

#### PCB 包含的信息

| 信息类别 | 内容 |
|---------|------|
| **进程标识信息** | 进程 ID、父进程 ID、用户 ID |
| **处理机状态** | 通用寄存器、PC、PSW 等（进程切换时保存/恢复） |
| **进程调度信息** | 进程状态、优先级、调度所需信息、等待事件 |
| **进程控制信息** | 程序和数据地址、资源清单、链接指针 |

#### 进程的组织方式

| 方式 | 说明 |
|------|------|
| **链接方式** | 将同一状态的 PCB 组织成一个队列（就绪队列、阻塞队列等） |
| **索引方式** | 建立索引表，指向相应状态的 PCB |

### 2.1.4 进程控制 ⭐

#### 进程的创建

```c
// 创建原语的主要工作
1. 申请空白 PCB
2. 为新进程分配资源
3. 初始化 PCB
4. 将新进程插入就绪队列
```

**引起进程创建的事件：** 用户登录、作业调度、提供服务（如创建子进程）、应用请求。

#### 进程的终止

```c
// 终止原语的主要工作
1. 根据被终止进程的标识符，找到该进程的 PCB
2. 若进程正在执行，立即终止该进程，置调度标志为真
3. 将其所有子孙进程终止
4. 归还资源给父进程或操作系统
5. 将 PCB 从所在队列中移除
```

#### 进程的阻塞与唤醒

**阻塞原语：** 运行态 → 阻塞态
- 找到要被阻塞进程的 PCB
- 保护现场（将 CPU 状态保存到 PCB）
- 将进程状态设为阻塞态
- 将 PCB 插入相应事件的等待队列
- 转进程调度

**唤醒原语：** 阻塞态 → 就绪态
- 从等待队列中找到相应进程的 PCB
- 将进程状态从阻塞态改为就绪态
- 将 PCB 从等待队列移除，插入就绪队列

⚠️ **阻塞是进程自身调用阻塞原语主动行为，唤醒是由其他进程（如 I/O 完成的中断处理程序）调用唤醒原语完成的。**

### 2.1.5 进程的通信 ⭐

| 方式 | 说明 | 特点 |
|------|------|------|
| **共享存储** | 多个进程共享一块内存区域 | 速度快，需要同步机制 |
| **消息传递** | 进程间通过发送/接收消息通信 | 直接/间接通信，使用方便 |
| **管道通信** | 利用管道文件（pipe）进行通信 | 半双工，先进先出 |
| **信号量** | 通过 P、V 操作进行同步和通信 | 经典同步机制 |

⚠️ **共享存储分为：**
- 基于数据结构的共享（低级）
- 基于存储区的共享（高级）

⚠️ **消息传递分为：**
- **直接通信：** send(P, msg)，receive(Q, msg)
- **间接通信（信箱）：** 通过信箱间接通信

### 2.1.6 线程 ⭐⭐

#### 线程的基本概念

线程是 CPU 调度的基本单位。一个进程可以包含多个线程，各线程共享进程的资源（地址空间、全局变量、文件等），但拥有各自的寄存器和栈。

#### 线程 vs 进程 ⭐

| 对比维度 | 进程 | 线程 |
|---------|------|------|
| 调度单位 | 是（传统OS） | 是（现代OS中） |
| 拥有资源 | 有 | 无（共享进程资源） |
| 并发性 | 进程间并发 | 线程间并发（更轻量） |
| 系统开销 | 大（创建/切换开销大） | 小 |
| 地址空间 | 独立 | 共享进程地址空间 |
| 通信 | 需要特殊机制 | 可直接读写共享变量 |

#### 线程的实现方式

| 类型 | 说明 | 调度 |
|------|------|------|
| **用户级线程（ULT）** | 线程在用户空间实现，内核不知道线程存在 | 由用户态线程库调度 |
| **内核级线程（KLT）** | 线程由内核管理 | 由内核调度 |

⚠️ **线程模型：**

| 模型 | 说明 |
|------|------|
| **多对一** | 多个用户线程映射到一个内核线程；一个线程阻塞则整个进程阻塞 |
| **一对一** | 每个用户线程对应一个内核线程；并发性好，但开销大 |
| **多对多** | n 个用户线程映射到 m 个内核线程（n ≥ m）；折中方案 |

## 2.2 处理机调度 ⭐⭐

### 2.2.1 调度的层次

| 层次 | 调度对象 | 频率 | 主要功能 |
|------|---------|------|---------|
| **高级调度（作业调度）** | 作业 | 低 | 从外存后备队列中选择作业调入内存 |
| **中级调度（内存调度）** | 进程（挂起态） | 中 | 将暂时不能运行的进程调至外存（挂起） |
| **低级调度（进程调度）** | 进程/线程 | 高 | 从就绪队列中选择一个进程分配 CPU |

⚠️ **三级调度的关系：**
- 高级调度决定哪些作业进入内存（建立进程）
- 中级调度决定哪些进程挂起到外存
- 低级调度决定哪个就绪进程获得 CPU

### 2.2.2 进程调度的时机

**可以调度的情况：**
1. 当前运行进程主动放弃 CPU（正常终止、异常终止、阻塞请求 I/O）
2. 当前运行进程被动放弃 CPU（时间片用完、更高优先级进程到达）

**不能调度的情况：**
1. 在处理中断的过程中
2. 进程在操作系统内核程序临界区中
3. 原子操作（如加锁/解锁）过程中

⚠️ **临界区 vs 内核临界区：**
- **内核临界区：** 访问操作系统内核数据结构的临界区，调度程序不能切换进程
- **普通临界区：** 访问普通共享资源的临界区，调度程序可以切换进程（但该资源仍被锁定）

### 2.2.3 进程调度的方式

| 方式 | 说明 |
|------|------|
| **非剥夺方式（非抢占式）** | 进程获得 CPU 后一直运行到完成或阻塞 |
| **剥夺方式（抢占式）** | 更高优先级进程到达时可抢占 CPU |

### 2.2.4 调度算法 ⭐⭐⭐（核心考点）

#### （1）先来先服务（FCFS）

按进程到达的先后顺序进行调度。

| 优点 | 缺点 |
|------|------|
| 公平、实现简单 | 对长作业有利，对短作业不利 |
| 有利于 CPU 繁忙型作业 | **平均等待时间长**，可能导致"饥饿"（convoy effect） |

⚠️ **FCFS 是非抢占式的。**

#### （2）短作业优先（SJF）⭐

选择估计运行时间最短的作业优先执行。

⚠️ **SJF 的特点：**
- **平均等待时间最短**（在非抢占式调度算法中）
- 对长作业不利，可能导致**饥饿**
- 实际中难以准确估计运行时间

**抢占式 SJF（SRTN/SRTF/最短剩余时间优先）：** 当新的进程到达时，比较当前进程的剩余时间和新进程的运行时间，选择较短的执行。

#### （3）优先级调度算法 ⭐

为每个进程分配一个优先级，优先级最高的进程先执行。

| 分类 | 说明 |
|------|------|
| 非抢占式 | 一旦运行就执行到完成或阻塞 |
| 抢占式 | 更高优先级到达时抢占 |

⚠️ **优先级的确定：**
- **静态优先级：** 创建时确定，运行时不变
- **动态优先级：** 运行过程中可调整

⚠️ **常见优先级设置规则：**
- 系统进程 > 用户进程
- 交互型进程 > 非交互型进程（I/O 繁忙型 > CPU 繁忙型）
- 可以用**老化（Aging）** 技术解决饥饿问题（逐渐提高等待时间长的进程的优先级）

#### （4）时间片轮转（RR）⭐⭐

按 FCFS 排列就绪队列，每个进程分配一个时间片（time quantum），时间片用完后将进程放到队尾。

⚠️ **时间片大小的影响：**
- **时间片太大：** 退化为 FCFS
- **时间片太小：** 频繁切换，系统开销大
- **合适的时间片：** 略大于一次典型的交互所需时间

⚠️ **RR 是抢占式的，响应时间好，适合分时系统。**

#### （5）多级反馈队列（MLFQ）⭐⭐

```
就绪队列1（优先级最高，时间片最小）
    ↓ （时间片用完降级）
就绪队列2（优先级次高，时间片较大）
    ↓
...
    ↓
就绪队列n（优先级最低，时间片最大/FCFS）
```

**规则：**
1. 新进程进入最高优先级队列1
2. 在队列 i 中用完时间片后，降到队列 i+1
3. 仅当队列 1~(i-1) 都为空时，才调度队列 i 中的进程
4. 新到达的高优先级进程可以抢占低优先级队列中的进程

⚠️ **MLFQ 的特点：**
- 终端型作业（短作业）：在高优先级队列完成，响应时间快
- 批处理作业（长作业）：逐级降级，最终在最低队列
- 繁忙型作业和 I/O 型作业都能得到较好的处理
- **综合了 FCFS、SJF、RR 和优先级调度的优点**

#### （6）多级队列调度

将就绪队列分成多个独立队列，每个队列有自己的调度算法。不同队列之间可以采用固定优先级或时间片划分。

## 2.3 同步与互斥 ⭐⭐⭐

### 2.3.1 同步与互斥的基本概念

**临界资源：** 一次仅允许一个进程使用的资源（如打印机、共享变量）。

**临界区：** 进程中访问临界资源的代码段。

#### 临界区访问原则（同步机制应遵循的准则）⭐

| 准则 | 说明 |
|------|------|
| **空闲让进** | 临界区空闲时，应允许一个请求进入临界区的进程立即进入 |
| **忙则等待** | 已有进程进入临界区时，其他试图进入的进程必须等待 |
| **有限等待** | 进程应在有限时间内进入临界区，防止"饥饿" |
| **让权等待** | 进程不能进入临界区时，应立即释放 CPU，防止"忙等" |

#### 互斥的软件实现方法

| 方法 | 说明 | 是否满足四准则 |
|------|------|-------------|
| 单标志法 | turn 变量 | 不满足空闲让进 |
| 双标志先检查法 | flag[] 数组 | 不满足忙则等待 |
| 双标志后检查法 | flag[] 数组 | 不满足有限等待 |
| Peterson 算法 ⭐ | turn + flag[] | 满足前三个，不满足让权等待 |

#### Peterson 算法 ⭐

```c
// 进程 Pi
flag[i] = true;       // 表示我想进入
turn = j;             // 谦让给对方
while (flag[j] && turn == j);  // 等待
// 临界区
flag[i] = false;      // 退出
```

#### 互斥的硬件实现方法

| 方法 | 说明 |
|------|------|
| **中断屏蔽方法** | 关中断 → 临界区 → 开中断 |
| **TestAndSet（TSL）** | 原子操作：读出旧值并设为 true |
| **Swap（XCHG）** | 原子操作：交换两个变量的值 |

⚠️ **硬件方法的缺点：** 不满足"让权等待"原则（忙等）。

### 2.3.2 信号量机制 ⭐⭐⭐

#### 整型信号量

```c
// wait 操作（P 操作）
wait(S) {
    while (S <= 0);  // 忙等
    S--;
}

// signal 操作（V 操作）
signal(S) {
    S++;
}
```

#### 记录型信号量 ⭐（408标准答案）

```c
typedef struct {
    int value;              // 资源数目
    struct process *L;      // 等待队列
} semaphore;

// P 操作
void wait(semaphore S) {
    S.value--;
    if (S.value < 0) {
        add this process to S.L;
        block();            // 阻塞进程，让出 CPU
    }
}

// V 操作
void signal(semaphore S) {
    S.value++;
    if (S.value <= 0) {
        remove a process P from S.L;
        wakeup(P);          // 唤醒进程
    }
}
```

⚠️ **信号量的物理含义：**
- `S.value ≥ 0`：表示可用资源数
- `S.value < 0`：`|S.value|` 表示等待该资源的进程数

#### 信号量实现同步和互斥 ⭐

**互斥（mutex）：**
```c
semaphore mutex = 1;
P(mutex);
// 临界区
V(mutex);
```

**同步（synchronization）：** 保证执行的先后顺序
```c
// 要求 S1 先执行，S2 后执行
semaphore S = 0;

// S1
代码段1;
V(S);

// S2
P(S);
代码段2;
```

### 2.3.3 经典同步问题 ⭐⭐⭐

#### （1）生产者-消费者问题 ⭐⭐

```c
semaphore mutex = 1;    // 互斥访问缓冲区
semaphore empty = n;    // 空闲缓冲区数
semaphore full = 0;     // 已用缓冲区数

// 生产者
producer() {
    while (true) {
        produce an item;
        P(empty);       // 申请空闲缓冲区
        P(mutex);       // 进入临界区
        add item to buffer;
        V(mutex);       // 离开临界区
        V(full);        // 增加已用缓冲区数
    }
}

// 消费者
consumer() {
    while (true) {
        P(full);        // 申请已用缓冲区
        P(mutex);       // 进入临界区
        remove item from buffer;
        V(mutex);       // 离开临界区
        V(empty);       // 增加空闲缓冲区数
        consume the item;
    }
}
```

⚠️ **P 操作的顺序不能颠倒！** 如果先 P(mutex) 再 P(empty)，可能导致死锁。

#### （2）读者-写者问题 ⭐⭐

```c
semaphore rw_mutex = 1;  // 读写互斥
semaphore mutex = 1;      // 保护 readcount
int readcount = 0;         // 读者计数

// 写者
writer() {
    while (true) {
        P(rw_mutex);
        // 写操作
        V(rw_mutex);
    }
}

// 读者
reader() {
    while (true) {
        P(mutex);
        readcount++;
        if (readcount == 1)  // 第一个读者要锁住写者
            P(rw_mutex);
        V(mutex);

        // 读操作

        P(mutex);
        readcount--;
        if (readcount == 0)  // 最后一个读者释放锁
            V(rw_mutex);
        V(mutex);
    }
}
```

⚠️ **读者优先：** 有读者在读时，新读者可以直接进入；写者必须等待所有读者完成。

#### （3）哲学家进餐问题 ⭐

5 个哲学家围坐圆桌，每两人之间一根筷子，每人需要两根筷子才能进餐。

**解决方案：**
1. 最多允许 4 个哲学家同时就坐
2. 奇数号哲学家先拿左筷子，偶数号先拿右筷子
3. 只有当左右筷子都可用时才拿起

#### （4）吸烟者问题

三个吸烟者和一个供应者，每个吸烟者需要三种材料中的两种。

### 2.3.4 管程 ⭐

**管程：** 一种高级同步机制，将共享数据和对数据的操作封装在一起。

```c
monitor ProducerConsumer {
    condition full, empty;
    int count = 0;

    procedure insert(item) {
        if (count == N) wait(full);
        // 插入数据
        count++;
        signal(empty);
    }

    procedure remove() {
        if (count == 0) wait(empty);
        // 移除数据
        count--;
        signal(full);
    }
}
```

⚠️ **管程的特性：**
- 管程内的数据只能通过管程内的过程访问
- 每次只允许一个进程在管程内执行某个过程
- 条件变量用于同步

## 2.4 死锁 ⭐⭐⭐

### 2.4.1 死锁的概念

**死锁：** 多个进程因竞争资源而造成的一种互相等待的现象，若无外力作用，这些进程都无法向前推进。

### 2.4.2 死锁产生的必要条件（Coffman条件）⭐

| 条件 | 说明 |
|------|------|
| **互斥条件** | 资源只能被一个进程使用 |
| **不剥夺条件** | 资源不能被强行剥夺 |
| **请求并保持条件** | 进程持有一些资源的同时又请求新资源 |
| **循环等待条件** | 存在进程资源的循环等待链 |

⚠️ **四个条件同时满足才会发生死锁。破坏任何一个条件即可预防死锁。**

### 2.4.3 死锁的处理策略

| 策略 | 说明 |
|------|------|
| **预防死锁** | 破坏死锁的四个必要条件之一 |
| **避免死锁** | 动态分配资源时进行安全检查（银行家算法） |
| **检测死锁** | 允许死锁发生，定期检测并解除 |
| **解除死锁** | 检测到死锁后采取措施解除 |

#### （1）死锁预防

| 破坏条件 | 方法 |
|---------|------|
| 破坏互斥条件 | 使用 SPOOLing 技术（如打印机） |
| 破坏不剥夺条件 | 申请不到资源时释放已持有的资源 |
| 破坏请求并保持 | 一次申请所有资源 |
| 破坏循环等待 | 按编号顺序申请资源 |

#### （2）死锁避免——银行家算法 ⭐⭐⭐

**核心数据结构：**
- **Available：** 可利用资源向量
- **Max：** 最大需求矩阵
- **Allocation：** 已分配矩阵
- **Need：** 需求矩阵（Need = Max - Allocation）

**安全性算法 ⭐：**

```
1. 初始化 Work = Available, Finish[i] = false
2. 找一个满足 Finish[i]=false 且 Need[i] ≤ Work 的进程 i
3. 若找到，Work = Work + Allocation[i], Finish[i] = true, 回到步骤 2
4. 若所有 Finish[i] = true，系统安全；否则不安全
```

**银行家算法（资源请求算法）⭐：**

进程 Pᵢ 请求资源 Requestᵢ：
1. 若 Requestᵢ > Need[i]，报错（请求超过需求）
2. 若 Requestᵢ > Available，等待（资源不足）
3. 试探分配：
   - Available = Available - Requestᵢ
   - Allocation[i] = Allocation[i] + Requestᵢ
   - Need[i] = Need[i] - Requestᵢ
4. 执行安全性算法：
   - 若安全 → 正式分配
   - 若不安全 → 恢复数据，让进程等待

#### （3）死锁检测

利用**资源分配图**检测死锁：
- 消去所有不阻塞且不孤立的进程的边
- 若能消去所有边，则无死锁

⚠️ **死锁定理：** S 为死锁状态的充分条件是**资源分配图不可完全简化**。

#### （4）死锁解除

| 方法 | 说明 |
|------|------|
| **资源剥夺法** | 挂起某些死锁进程，剥夺资源给其他进程 |
| **撤销进程法** | 强制撤销部分或全部死锁进程 |
| **进程回退法** | 让死锁进程回退到某个安全状态 |

### 例题1：调度算法计算

### 例题3：信号量应用

semaphore S = 0;

P1() {

    V(S);

    P(S);
