# 第5章 传输层

## 5.1 传输层提供的服务 ⭐

### 5.1.1 传输层的功能

1. **提供进程间的逻辑通信**（端到端通信）
2. **复用和分用**
   - 复用：多个应用进程可以同时使用传输层服务
   - 分用：传输层把收到的信息分别交付给各应用进程
3. **差错检测**（首部和数据部分的校验）
4. 对收到的报文进行**差错检测**（IP 层只检验首部）

### 5.1.2 端口 ⭐

**端口号（16 位）：** 用于标识主机中的应用进程。

| 类型 | 范围 | 说明 |
|------|------|------|
| **熟知端口号** | 0 ~ 1023 | 分配给最重要的应用程序 |
| **登记端口号** | 1024 ~ 49151 | 为没有熟知端口号的应用程序使用 |
| **客户端端口号** | 49152 ~ 65535 | 临时端口号 |

**常见的熟知端口号 ⭐：**

| 应用层协议 | 传输层协议 | 端口号 |
|-----------|-----------|--------|
| FTP（数据） | TCP | 20 |
| FTP（控制） | TCP | 21 |
| SSH | TCP | 22 |
| TELNET | TCP | 23 |
| SMTP | TCP | 25 |
| DNS | UDP | 53 |
| DHCP（服务器） | UDP | 67 |
| DHCP（客户端） | UDP | 68 |
| HTTP | TCP | 80 |
| POP3 | TCP | 110 |
| HTTPS | TCP | 443 |

## 5.2 UDP 协议 ⭐⭐

### 5.2.1 UDP 概述

**用户数据报协议（UDP）：** 无连接的、不可靠的传输层协议。

### 5.2.2 UDP 的特点

| 特点 | 说明 |
|------|------|
| 无连接 | 发送数据前不需要建立连接 |
| 不可靠 | 尽最大努力交付，不保证不丢失、不重复 |
| 面向报文 | UDP 对应用层交下来的报文，既不合并也不拆分 |
| 没有拥塞控制 | 网络出现拥塞时不会降低发送速率 |
| 支持一对一、一对多、多对多通信 | |
| 首部开销小（8 字节） | |

### 5.2.3 UDP 首部格式 ⭐

```
| 16位源端口号 | 16位目的端口号 |
| 16位UDP长度  | 16位UDP校验和  |
|           数据（可选）         |
```

⚠️ **UDP 首部只有 8 字节。**

⚠️ **UDP 长度 = 首部 + 数据的总长度（最小值为 8）。**

### 5.2.4 UDP 校验 ⭐

**伪首部 + UDP 首部 + 数据** 一起进行校验。

伪首部包含：源 IP 地址、目的 IP 地址、协议号（17）、UDP 长度。

⚠️ **伪首部仅用于校验，不实际发送。**

## 5.3 TCP 协议 ⭐⭐⭐

### 5.3.1 TCP 概述

**传输控制协议（TCP）：** 面向连接的、可靠的、基于字节流的传输层协议。

### 5.3.2 TCP 的特点

| 特点 | 说明 |
|------|------|
| 面向连接 | 通信前必须建立连接 |
| 可靠传输 | 保证数据无差错、不丢失、不重复且有序 |
| 面向字节流 | TCP 把应用层数据看作无结构的字节流 |
| 全双工通信 | 双方都可以同时发送和接收 |
| 点对点 | 每一条 TCP 连接只能有两个端点 |

### 5.3.3 TCP 报文段格式 ⭐⭐

```
| 16位源端口号         | 16位目的端口号         |
| 32位序号（SEQ）                               |
| 32位确认号（ACK号）                           |
| 4位首部长度|6位保留| URG ACK PSH RST SYN FIN | 16位窗口大小 |
| 16位校验和           | 16位紧急指针            |
| 选项（可变）                                   |
| 数据                                          |
```

⚠️ **重要字段：**
- **序号（SEQ）：** 本报文段数据部分第一个字节的编号
- **确认号（ACK）：** 期望收到对方下一个报文段的第一个数据字节的编号（累计确认）
- **首部长度：** 以 4 字节为单位（最小 5，即 20 字节）
- **窗口：** 接收窗口大小（流量控制）

⚠️ **TCP 标志位：**

| 标志 | 含义 |
|------|------|
| **URG** | 紧急位，紧急指针有效 |
| **ACK** | 确认位，确认号有效 |
| **PSH** | 推送位，接收方应尽快交付应用层 |
| **RST** | 复位，TCP 连接出现严重差错 |
| **SYN** | 同步位，连接请求/接受报文 |
| **FIN** | 终止位，释放连接 |

### 5.3.4 TCP 连接管理 ⭐⭐⭐

#### 三次握手（建立连接）⭐⭐⭐

```
客户端                           服务器
   |                                |
   |──── SYN=1, SEQ=x ──────────→  |  第一次握手
   |                                |
   |←── SYN=1, ACK=1, SEQ=y,      |
   |     ACK=x+1 ────────────────  |  第二次握手
   |                                |
   |──── ACK=1, SEQ=x+1,         |
   |     ACK=y+1 ────────────────→ |  第三次握手
   |                                |
   |       连接建立                  |
```

⚠️ **为什么需要三次握手？**
- 防止已失效的连接请求报文段突然又传到了服务端，导致服务端错误地建立连接（浪费资源）
- 确保双方的初始序号（ISN）都被对方确认

#### 四次挥手（释放连接）⭐⭐⭐

```
客户端                           服务器
   |                                |
   |──── FIN=1, SEQ=u ──────────→  |  第一次挥手（客户端请求释放）
   |                                |
   |←── ACK=1, SEQ=v, ACK=u+1 ──  |  第二次挥手（服务器确认）
   |                                |
   |      （服务器继续发送剩余数据）    |
   |                                |
   |←── FIN=1, ACK=1, SEQ=w,      |
   |     ACK=u+1 ────────────────  |  第三次挥手（服务器请求释放）
   |                                |
   |──── ACK=1, SEQ=u+1,         |
   |     ACK=w+1 ────────────────→ |  第四次挥手（客户端确认）
   |                                |
   |      等待 2MSL 后关闭           |
```

⚠️ **为什么需要四次挥手？**
- TCP 是全双工的，每个方向的连接需要单独关闭
- 服务端收到 FIN 后可能还有数据需要发送

⚠️ **TIME_WAIT 状态（等待 2MSL）的原因：**
1. 确保最后一个 ACK 能到达服务端（若丢失，服务端会重传 FIN）
2. 确保本次连接的所有报文段都从网络中消失（防止与新连接混淆）

### 5.3.5 TCP 可靠传输 ⭐⭐

#### 确认机制

- **累积确认：** ACK 号表示期望接收的下一个字节的编号
- **延迟确认：** 接收方可以在收到数据后延迟 500ms 再发送确认

#### 超时重传 ⭐

- 发送方每发送一个报文段就启动一个**超时计时器（RTO）**
- 若在 RTO 内未收到确认，则重传该报文段
- RTO 根据 RTT（往返时延）动态调整

$$RTO = RTT_S + 4 \times RTT_D$$

其中 RTT_S 为加权平均 RTT，RTT_D 为 RTT 偏差的加权平均。

#### 冗余 ACK（快速重传）⭐⭐

接收方收到失序的报文段时，立即发送**重复确认**（冗余 ACK）。

**快速重传：** 发送方收到 **3 个重复确认**（共 4 个相同的 ACK），立即重传丢失的报文段（不等超时）。

### 5.3.6 TCP 流量控制 ⭐⭐

**流量控制：** 控制发送方的发送速率，使接收方来得及接收。

**滑动窗口机制：**

- 接收方在 ACK 中通告**接收窗口（rwnd）**的大小
- 发送方维护**发送窗口**，大小 ≤ rwnd
- 窗口为 0 时发送方停止发送，但可以发送 **1 字节的探测报文段**（防止死锁）

⚠️ **传输层的流量控制 vs 数据链路层的流量控制：**
- 传输层：端到端，使用可变大小的滑动窗口
- 数据链路层：结点到结点，窗口大小通常固定

### 5.3.7 TCP 拥塞控制 ⭐⭐⭐

**拥塞控制：** 防止过多的数据注入网络，避免网络中的路由器或链路过载。

⚠️ **拥塞窗口（cwnd）：** 发送方维护的控制发送速率的窗口。

**发送窗口 = min(cwnd, rwnd)**

#### 四种拥塞控制算法 ⭐⭐⭐

**（1）慢开始（Slow Start）**

- 初始 cwnd = 1 MSS（最大报文段长度）
- 每收到一个 ACK，cwnd + 1 MSS（即**每轮翻倍**）
- cwnd 增长到**慢开始门限（ssthresh）** 时转为拥塞避免

**（2）拥塞避免（Congestion Avoidance）**

- cwnd 每轮（RTT）增加 1 MSS（**线性增长**）
- 直到发生拥塞（超时）

**（3）快重传（Fast Retransmit）**

- 收到 3 个重复确认后立即重传（不等超时）

**（4）快恢复（Fast Recovery）**

- 发生快重传时：ssthresh = cwnd/2，cwnd = ssthresh，进入拥塞避免
- 发生超时时：ssthresh = cwnd/2，cwnd = 1，进入慢开始

#### 拥塞控制的过程 ⭐

```
cwnd
 ↑
 |          超时 ssthresh减半
 |         /  cwnd=1
 |        /     \
 |       /       \ 慢开始
 |      /  cwnd按指数增长  \
 |     /                    \ cwnd >= ssthresh
 |    /                      \ 拥塞避免（线性增长）
 |   /                         \
 |  / cwnd 指数增长               \
 | /                               \
 |/________ssthresh__________________\_____
 └─────────────────────────────────────→ 轮次
```

⚠️ **cwnd 变化总结：**
- 慢开始阶段：cwnd 指数增长（1, 2, 4, 8, 16, ...）
- 拥塞避免阶段：cwnd 线性增长（每 RTT +1）
- 超时：ssthresh = cwnd/2，cwnd = 1，重新慢开始
- 3 个重复 ACK：ssthresh = cwnd/2，cwnd = ssthresh，拥塞避免

## 5.4 本章例题

### 例题1：三次握手与序号

TCP 三次握手过程中，若客户端发送 SYN=1, SEQ=1000，服务器回应 SYN=1, ACK=1, SEQ=2000, ACK=？

**答案：** ACK = 1000 + 1 = **1001**

### 例题2：拥塞控制

设 ssthresh = 16，画出发生以下事件时 cwnd 的变化曲线：正常增长到 24，发生超时；增长到 20 时收到 3 个重复 ACK。

**解析：**

- cwnd 从 1 开始慢开始（1, 2, 4, 8, 16）
- cwnd 达到 16，转拥塞避免（17, 18, ...）
- 增长到 24 时超时：ssthresh = 12, cwnd = 1
- 重新慢开始（1, 2, 4, 8, 12）
- cwnd 达到 12，转拥塞避免（13, 14, ...）
- 增长到 20 时 3 个重复 ACK：ssthresh = 10, cwnd = 10
- 从 10 开始拥塞避免（11, 12, ...）

### 例题3：序号和确认号计算

TCP 连接中，发送方已发送序号为 100~199 的 100 字节数据，接收方正确收到。接收方的确认报文中 ACK 号应为多少？

**答案：** ACK = 199 + 1 = **200**

### 例题4：UDP vs TCP

DNS 使用什么传输层协议？为什么？

**答案：** DNS 使用 **UDP**（端口 53）。

原因：
- DNS 查询通常数据量小，UDP 头部开销小
- UDP 无连接，不需要建立和释放连接的开销
- DNS 有自己的超时重传机制
- 但在区域传送（大数据量）时使用 TCP
