# 第6章 应用层

## 6.1 网络应用模型 ⭐

### 6.1.1 客户/服务器模型（C/S）

| 角色 | 说明 |
|------|------|
| **服务器** | 提供服务，被动等待请求，永久运行 |
| **客户** | 请求服务，主动发起请求，间歇性运行 |

**特点：** 客户之间不直接通信，服务器有固定的 IP 地址和熟知端口号。

**典型应用：** Web（HTTP）、FTP、电子邮件。

### 6.1.2 P2P 模型（对等网络）

每个结点既是服务器又是客户，各结点地位平等。

**优点：** 可扩展性好、健壮性好。
**缺点：** 管理困难。

**典型应用：** BitTorrent、eMule。

## 6.2 DNS（域名系统）⭐⭐

### 6.2.1 DNS 的概述

**DNS：** 将域名解析为 IP 地址的分布式数据库系统。

⚠️ **DNS 使用 UDP，端口号 53。**

### 6.2.2 层次域名空间 ⭐

```
         根（.）
        /   \
      com    edu     cn     ...
      /       |       \
   www    mit.edu    baidu.cn
          /           \
      web.mit.edu   www.baidu.cn
```

| 域名层级 | 例子 |
|---------|------|
| 顶级域名 | com, edu, gov, org, cn, us |
| 二级域名 | baidu.com, tsinghua.edu.cn |
| 三级域名 | www.baidu.com |

### 6.2.3 域名服务器

| 类型 | 说明 |
|------|------|
| **根域名服务器** | 最高层次，知道所有顶级域名服务器的地址 |
| **顶级域名服务器** | 管理该顶级域名下注册的二级域名 |
| **权限域名服务器** | 负责一个区的域名解析 |
| **本地域名服务器** | 主机的 DNS 查询首先发给本地域名服务器 |

### 6.2.4 域名解析过程 ⭐⭐

#### 递归查询

主机向本地 DNS 服务器递归查询，本地 DNS 服务器代替主机向其他服务器递归查询。

#### 迭代查询 ⭐（常用）

```
主机 → 本地DNS → 根DNS：返回顶级DNS地址
本地DNS → 顶级DNS → 权限DNS：返回最终IP地址
本地DNS → 主机：返回IP地址
```

⚠️ **实际中常用：主机→本地 DNS（递归），本地 DNS→其他 DNS（迭代）。**

### 6.2.5 高速缓存

本地域名服务器维护高速缓存，存放最近解析过的域名到 IP 的映射。

⚠️ **高速缓存中的映射有生存时间（TTL），超时后自动删除。**

## 6.3 FTP（文件传输协议）⭐

### 6.3.1 FTP 概述

FTP 使用两个并行的 TCP 连接：

| 连接 | 端口 | 用途 |
|------|------|------|
| **控制连接** | 21 | 传输控制命令（端口号、用户名、密码等） |
| **数据连接** | 20（主动模式） | 传输文件数据 |

⚠️ **控制连接在整个会话期间保持，数据连接在每次传输时建立和释放。**

⚠️ **FTP 是有状态的协议**（服务器保存用户的状态信息）。

### 6.3.2 FTP 工作模式

| 模式 | 说明 |
|------|------|
| **主动模式（PORT）** | 服务器主动向客户端发起数据连接（服务器端口 20） |
| **被动模式（PASV）** | 客户端向服务器发起数据连接（服务器随机端口） |

## 6.4 电子邮件 ⭐

### 6.4.1 电子邮件系统的组成

```
发件人 → 用户代理 → 邮件服务器(SMTP) → 邮件服务器 → 用户代理 → 收件人
         (UA)        ↑        (SMTP/POP3/IMAP)        (UA)
                    SMTP
```

**三个主要组成：**
1. **用户代理（UA）：** 邮件客户端软件
2. **邮件服务器：** 发送和接收邮件
3. **协议：** SMTP（发送）、POP3/IMAP（接收）

### 6.4.2 SMTP（简单邮件传输协议）⭐

- 使用 **TCP，端口号 25**
- 基于 C/S 模式
- 只能传输 7 位 ASCII 码

⚠️ **SMTP 的缺点：** 不能传送可执行文件或其他二进制对象，不能传送非 ASCII 码数据。

**MIME（通用因特网邮件扩充）：** 解决 SMTP 的局限性，在邮件首部增加 MIME 首部，对非 ASCII 数据进行编码。

### 6.4.3 POP3（邮局协议版本 3）⭐

- 使用 **TCP，端口号 110**
- 下载并删除模式 / 下载并保留模式

### 6.4.4 IMAP（因特网报文存取协议）

- 使用 **TCP，端口号 143**
- 邮件保留在服务器上，可以在服务器上管理邮件

### 6.4.5 基于万维网的电子邮件

使用 HTTP 代替 SMTP（发送）和 POP3（接收）。

⚠️ **邮件服务器之间仍使用 SMTP 传输邮件。**

## 6.5 WWW（万维网）⭐⭐

### 6.5.1 WWW 的概念

万维网是一个大规模的、联机式的信息储藏所，通过 URL 标识资源，通过 HTTP 协议传送资源，通过 HTML 描述资源。

### 6.5.2 URL（统一资源定位符）⭐

```
<协议>://<主机>:<端口>/<路径>
```

例：`http://www.example.com:80/index.html`

### 6.5.3 HTTP（超文本传输协议）⭐⭐⭐

#### HTTP 概述

- 使用 **TCP，端口号 80**（HTTPS 使用 443）
- HTTP 是**无状态的**协议（服务器不保存客户端状态）
- 使用 **Cookie** 机制跟踪用户状态
- 支持**持久连接**和**非持久连接**

#### 持久连接 vs 非持久连接 ⭐

| 类型 | 说明 | TCP 连接数 |
|------|------|-----------|
| **非持久连接（HTTP/1.0）** | 每个对象需要单独建立 TCP 连接 | 每个对象 1 次 |
| **持久连接（HTTP/1.1 默认）** | 一个 TCP 连接可以传输多个对象 | 整个页面 1 次 |

⚠️ **持久连接分为：**
- **非流水线方式：** 收到响应后才能发送下一个请求
- **流水线方式 ⭐：** 可以连续发送请求（HTTP/1.1 默认）

#### HTTP 报文格式 ⭐⭐

**请求报文：**
```
请求行（方法 URL 版本）
首部行
空行
实体主体（可选）
```

**常见请求方法：**

| 方法 | 说明 |
|------|------|
| **GET** | 请求读取 URL 标识的信息 |
| **POST** | 向服务器提交数据 |
| **HEAD** | 类似 GET，但只返回首部，不返回主体 |
| **PUT** | 上传文件 |
| **DELETE** | 删除文件 |

**响应报文：**
```
状态行（版本 状态码 短语）
首部行
空行
实体主体
```

**常见状态码 ⭐：**

| 类别 | 说明 | 例子 |
|------|------|------|
| **1xx** | 通知信息 | 100 Continue |
| **2xx** | 成功 | **200 OK** |
| **3xx** | 重定向 | 301 永久移动, 302 临时移动, 304 未修改 |
| **4xx** | 客户端错误 | **400 Bad Request**, **403 Forbidden**, **404 Not Found** |
| **5xx** | 服务器错误 | **500 Internal Server Error**, **503 Service Unavailable** |

#### HTTP 的持续时间计算 ⭐

**例：** 一个 HTML 页面引用 1 个图片和 1 个 CSS 文件。RTT = 100ms，HTML 文件传输时间 200ms，图片传输时间 100ms，CSS 传输时间 50ms。

**非持久连接（无并行）：**
- TCP 建立：1 RTT
- HTML 请求和响应：1 RTT + 200ms
- 图片 TCP 建立 + 请求响应：2 RTT + 100ms
- CSS TCP 建立 + 请求响应：2 RTT + 50ms
- 总时间 = 1 + 3×2 = 7 RTT + 350ms = 1050ms

**持久连接（流水线）：**
- TCP 建立：1 RTT
- HTML 请求和响应：1 RTT + 200ms
- 图片和 CSS 请求：可以连续发出
- 图片响应：1 RTT + 100ms
- CSS 响应：在图片之后（流水线），但服务器可以并行处理
- 总时间 ≈ 1 + 1 + 1 + 1 = 4 RTT + 350ms = 750ms

⚠️ **流水线方式下，所有对象的请求可以一次性发出。**

## 6.6 本章例题

### 例题1：DNS 解析过程

简述使用迭代查询方式解析域名 `www.example.com` 的过程。

**答案：**

1. 主机向本地 DNS 服务器发送 DNS 查询请求
2. 本地 DNS 服务器查询缓存，若没有则向根 DNS 服务器查询
3. 根 DNS 服务器返回 `.com` 顶级域名服务器的地址
4. 本地 DNS 服务器向 `.com` 顶级域名服务器查询
5. 顶级域名服务器返回 `example.com` 权限域名服务器的地址
6. 本地 DNS 服务器向 `example.com` 权限域名服务器查询
7. 权限域名服务器返回 `www.example.com` 的 IP 地址
8. 本地 DNS 服务器将结果返回给主机并缓存

### 例题2：HTTP 与 TCP 的关系

HTTP/1.0 和 HTTP/1.1 在 TCP 连接使用上有何不同？这对网页加载时间有何影响？

**答案：**

- HTTP/1.0 使用非持久连接：每个对象（HTML、图片、CSS 等）都需要单独建立和释放 TCP 连接，增加了 RTT 开销
- HTTP/1.1 使用持久连接（默认流水线方式）：一个 TCP 连接可以传输多个对象，减少了建立和释放连接的开销
- HTTP/1.1 可以显著减少网页加载时间，特别是当页面包含很多小对象时

### 例题3：邮件传输过程

用户 A 向用户 B 发送一封电子邮件，简述整个过程。

**答案：**

1. A 的用户代理使用 SMTP 将邮件发送到 A 的邮件服务器
2. A 的邮件服务器使用 SMTP 将邮件传送到 B 的邮件服务器
3. B 的用户代理使用 POP3/IMAP 从 B 的邮件服务器取回邮件
4. B 阅读邮件

⚠️ **SMTP 用于"推"（发送方 → 邮件服务器 → 接收方邮件服务器），POP3/IMAP 用于"拉"（用户代理 ← 邮件服务器）。**

### 例题4：HTTP 状态码

浏览器访问某网页时收到状态码 404，是什么意思？收到 301 呢？

**答案：**
- **404 Not Found：** 请求的资源不存在
- **301 Moved Permanently：** 资源已永久移动到新的 URL，浏览器应自动重定向
